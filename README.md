# Account Health Early Warning

A working prototype of an AI-assisted account health scan for customer success teams. Synthetic account data is scored by Claude against a versioned scoring prompt, producing a weekly risk tier, reasoning, and a recommended next action with an owner.

Built by Nancy Tu to demonstrate how CS judgment gets encoded into an AI system. The scoring prompt is visible inside the demo.

## What it shows

- Portfolio-level triage. At risk, watch, healthy
- Per-account assessment with the specific signals that drove the tier
- A next action with a named owner, because a warning without an owner is noise
- The full scoring prompt, versioned, so the judgment is auditable

## Stack

Single static HTML file (`account-health-demo.html`). No build step, no dependencies. Assessments in this demo were generated by Claude (claude-sonnet-4-6) from the synthetic telemetry.

## Run it

Download `account-health-demo.html` and open it in any browser. That is the whole setup.

Optional hosted version. Enable GitHub Pages in this repo (Settings, then Pages, select the main branch) and the demo becomes a public URL with no extra tooling.

## Using this with real data

The demo hardcodes eight synthetic accounts inside one marked block at the top of the script section. To run this against a real customer portfolio, the dashboard needs four data inputs per account and one scoring pipeline.

### Data inputs

1. **Usage telemetry.** Weekly product usage volume for the last 8 weeks, per account. Source depends on the stack. A product analytics tool (Amplitude, PostHog, Mixpanel), a data warehouse (Snowflake, BigQuery), or the product database itself.
2. **Support ticket excerpts.** Subject lines and recent message text from the last 30 days. Source is the helpdesk (Zendesk, Intercom, Plain, or a shared Slack channel export).
3. **Touchpoint history.** Date and attendees of the last meaningful interaction. Source is the CRM (HubSpot, Salesforce, Attio) or calendar data.
4. **Contract context.** Plan tier, contract stage, renewal date. Source is the CRM or the billing system.

### Scoring pipeline

A scheduled job (a cron script, a GitHub Action, or an automation platform) runs weekly and does four things.

1. Pull the four inputs above for each account and assemble one JSON object per account
2. Send each account object to the Claude API together with the scoring prompt shown in the demo
3. Collect the returned assessments (tier, reasoning, signals, next action, owner) into a single `accounts.json` file
4. Publish `accounts.json` next to the dashboard file

### Wiring the dashboard

Replace the hardcoded `accounts` array in the script block with a fetch call that loads `accounts.json` at page load. The rendering code stays identical. Everything the dashboard displays is already shaped like the JSON the scoring prompt returns.

### Steps to go live

1. Decide the four data sources for your stack and confirm API or export access to each
2. Write the weekly pull script and shape one JSON object per account
3. Store the scoring prompt in the repo as a versioned file so every assessment is traceable to a prompt version
4. Add the Claude API call (an Anthropic API key and roughly 20 lines of code)
5. Schedule the job weekly, publish `accounts.json`, and point the dashboard at it
6. Route red-tier accounts to a Slack alert so the warning reaches the owner without anyone opening the dashboard

A realistic first version of this pipeline is one to two days of work for a portfolio under 100 accounts. The judgment work, deciding which signals matter and encoding them into the prompt, is the part that takes CS experience rather than engineering time.
